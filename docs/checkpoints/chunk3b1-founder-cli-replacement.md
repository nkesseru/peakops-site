# Chunk 3B-1 — Founder-CLI Replacement Checkpoint

**Branch:** `chunk3/b1-founder-cli-replacement`
**Prepared:** 2026-06-22
**Goal:** Replace the 5-step manual founder-only CLI dance for provisioning a new customer org with a single CLI command runnable by a non-engineer Customer Success person.

This is the **smallest, highest-leverage** slice of Chunk 3B (the broader 4-day "activation operations" sprint). Out-of-scope items deferred to follow-on chunks.

---

## A. Executive Summary

### What this delivers

Before Chunk 3B-1, provisioning a single new customer org required Nick to personally execute (per the Chunk 3A audit):

1. `node setInternalAdminClaim.cjs --target-email=… --apply` (founder claim mint, one-time)
2. Create Firebase Auth user for the customer admin via Firebase Console (manual GUI step)
3. `curl POST /bootstrapPilotOrgV1` with the ownerUid + name + industry + timezone
4. `node setClaims.cjs <ownerUid> <orgId> owner --apply` (custom claims mint)
5. Manually compose and send a welcome email containing the customer's first-login URL

…and steps 2 + 4 + 5 then **repeated per teammate**.

After Chunk 3B-1, the same provisioning is one command:

```bash
node scripts/activateCustomerOrg.cjs \
  --name="Butler America Telecom" \
  --industry=telecom \
  --admin-email=admin@butleramerica.com \
  --admin-name="Sarah Butler" \
  --teammate=field1@butleramerica.com:field \
  --teammate=sup1@butleramerica.com:supervisor \
  --apply
```

The script outputs an org ID, every magic link, and a copy-pasteable welcome email template. **No Firebase Console, no Bash chaining, no `setClaims.cjs` per teammate.**

### Findings fixed

| Audit finding | Resolution |
|---|---|
| **CRITICAL — No `createOrgV1` callable** (only the engineer-facing `bootstrapPilotOrgV1`, which requires `ownerUid` and does not mint claims or create the Auth user) | New `functions_clean/createOrgV1.js`. POST { orgName, industry, ownerEmail, ownerName?, timezone? } → atomic: derive orgId from slug + find-or-create Auth user + mint owner claims + write org doc + write member doc + audit row + return `firstLoginUrl`. peakopsInternalAdmin gate. Idempotent. |
| **CRITICAL — No `inviteOrgMemberV1` callable** ("real invite-send pipeline drafts only" per OnboardingClient header comment) | New `functions_clean/inviteOrgMemberV1.js`. POST { orgId, email, role, displayName? } → atomic: find-or-create Auth user + merge claims (append orgId to `orgIds` array, set role) + write member doc + audit row + return magic link. Caller must be admin/owner of orgId (or peakopsInternalAdmin). Idempotent. Rejects role=owner. Refuses implicit role-change for existing members. |
| **CRITICAL — `setClaims.cjs` is the only role-assignment surface** (per-user CLI; not scalable to 25+ teammates across 5 customers) | The two new callables both call `setCustomUserClaims` atomically as part of org/invite provisioning. `setClaims.cjs` remains available for backward-compat support cases (e.g., grandfathering legacy users), but is no longer required for normal provisioning. |
| **CRITICAL — No CS-runnable activation script** | New `scripts/activateCustomerOrg.cjs`. Argparse with --name / --industry / --admin-email / --admin-name? / --timezone? / --teammate=email:role (repeatable) / --apply (default off). Dry-run shows the plan; apply executes via Bearer-token POST to both callables. Outputs orgId, every magic link, email template. |

- **Custom archetype self-service** — deferred per Chunk 3A audit.
- **NORS / BABA / SAR rulepack content** — deferred per audit.

## C. Evidence

### Fix 1 — `createOrgV1` callable

| Item | Detail |
|---|---|
| **Files added** | `functions_clean/createOrgV1.js` (291 lines); register entry in `functions_clean/index.js` |
| **Why added** | The audit identified no `createOrgV1` callable. `bootstrapPilotOrgV1` exists but requires an existing Auth user (`ownerUid`) and does not mint claims — leaving steps 2 and 4 of the manual dance unsolvable. |
| **What it does atomically** | Validates input → derives orgId from `toCustomerSlug(orgName)` → checks idempotency → find-or-create Auth user via `admin.auth().getUserByEmail` / `createUser` → mints custom claims `{ orgId, role: "owner", orgIds: [orgId] }` (merging onto existing claims so `peakopsInternalAdmin` is preserved) → generates first-login URL via `generatePasswordResetLink` → writes org doc + owner member doc + `ORG_CREATED` audit row in a single Firestore batch → returns `{ orgId, ownerUid, ownerEmail, firstLoginUrl, authUserCreated, already }`. |
| **Auth gate** | `peakopsInternalAdmin` claim (same as `bootstrapPilotOrgV1`). Founder-only for now; a CS-delegatable wrapper UI is out-of-scope for 3B-1. |
| **Idempotency** | Re-calling with the same orgName returns `{ already: true, orgId, ownerUid }` and writes nothing. |
| **Error surface** | Structured 4xx for input validation (`invalid_industry`, `invalid_ownerEmail`, `orgName_unsluggable`, `reserved_orgId_prefix`), 502 for downstream failures (`auth_user_provision_failed`, `claims_mint_failed`), 500 for unhandled. |
| **Verified via** | `scripts/dev/test_create_org_invite_member_wired.mjs` (9 assertions on this file); module requires cleanly in pure Node. |

### Fix 2 — `inviteOrgMemberV1` callable

| Item | Detail |
|---|---|
| **Files added** | `functions_clean/inviteOrgMemberV1.js` (294 lines); register entry in `functions_clean/index.js` |
| **Why added** | The audit identified that the onboarding wizard's Team step writes "drafts only — no email is sent." There was no real invitation surface. |
| **What it does atomically** | Validates orgId + email + role → caller must be admin/owner of orgId (`assertActorMember` + `ROLES_ADMIN_ONLY`) — or `peakopsInternalAdmin` for founder bootstrap — → find-or-create Auth user → merges claims (APPENDS orgId to `orgIds` array; sets `role` + singular `orgId` to the newly-invited org) → checks role conflict if member already exists → writes member doc with role-based default permissions + `MEMBER_INVITED` audit row in a single batch → generates magic link → returns `{ uid, email, role, magicLink, authUserCreated, already }`. |
| **Multi-org behavior** | Existing `orgIds` are preserved; new orgId is appended without duplicates. This lets a user belong to multiple orgs. |
| **Idempotency** | If member exists with same role + active status: `{ already: true }`. If member exists with different role: 409 `role_conflict` (refuses implicit role change). |
| **Rejects role="owner"** | Owner is set exclusively by `createOrgV1`. Invite endpoint accepts admin/supervisor/field/viewer. |
| **Verified via** | `scripts/dev/test_create_org_invite_member_wired.mjs` (10 assertions); module requires cleanly. |

### Fix 3 — `scripts/activateCustomerOrg.cjs`

| Item | Detail |
|---|---|
| **Files added** | `scripts/activateCustomerOrg.cjs` (264 lines) |
| **Why added** | CS-runnable wrapper around the two new callables. Replaces ~5 CLI invocations + ~3 Console clicks per customer with one command. |
| **Arg surface** | `--name`, `--industry`, `--admin-email`, `--admin-name?`, `--timezone?`, `--teammate=email:role` (repeatable), `--apply` (default off). |
| **Default behavior** | DRY-RUN. Prints the activation plan + the derived orgId + every callable that would be invoked. Writes nothing. |
| **Apply behavior** | Calls `/createOrgV1` first, then `/inviteOrgMemberV1` per teammate. Partial-failure tolerant (continues invitations even if one teammate fails). Returns non-zero exit code on partial failure so it surfaces in CI. |
| **Output** | Per-step HTTP status + body summary, then a summary table (orgId, every magic link), then a copy-pasteable welcome email template. |
| **Auth** | Requires `FIREBASE_ID_TOKEN` env var with a token from a `peakopsInternalAdmin:true` user. |
| **Smoke-tested** | `--help` renders correctly; dry-run with full args prints the plan and exits 0 without touching any backend. |

## D. Deployment Notes

### Risk level
**LOW.** All changes are additive:
- New Cloud Functions; no signature changes to existing callables.
- No schema changes (`orgs/{orgId}` and `orgs/{orgId}/members/{uid}` doc shapes already exist from `bootstrapPilotOrgV1`; both new callables write the same fields).
- No Firestore rule changes.
- No Storage rule changes.
- Idempotent: re-running the activate script with the same `--name` returns `{ already: true }` and writes nothing.

### Deploy lanes

1. **Cloud Functions** — both new callables (`createOrgV1`, `inviteOrgMemberV1`) plus the updated `index.js`. Run `firebase deploy --only functions --project peakops-pilot`. No app/Vercel deploy required.

That's it. Single-lane deploy. No coordination required.

### Rollback plan

If either callable misbehaves in production:

```bash
# Revert the merge commit
git revert <PR merge SHA>
git push origin main

# Redeploy Cloud Functions (drops createOrgV1 + inviteOrgMemberV1 from the runtime)
firebase deploy --only functions --project peakops-pilot
```

The activate script becomes a no-op (both POSTs would 404). The pre-3B-1 manual founder flow (setInternalAdminClaim + setClaims + bootstrapPilotOrgV1) still works as it did before this PR.

Pre-deploy (any branch):

Post-deploy:
- [ ] **Smoke a fresh provisioning** — pick a unique org name (e.g., "Chunk 3B-1 Smoke Co"). Run with `--apply`. Expected:
  - HTTP 200 from `/createOrgV1`
  - Returned `firstLoginUrl` is a valid `https://app.peakops.app/auth/action?...` URL
  - Firestore: `orgs/chunk-3b-1-smoke-co` exists with `kind: "customer"`, `status: "active"`, `ownerUserId` set
  - Firestore: `orgs/chunk-3b-1-smoke-co/members/<ownerUid>` exists with `role: "owner"`, `status: "active"`
  - Firestore: `orgs/chunk-3b-1-smoke-co/audit/create_org_*` row exists
  - Firebase Auth: a user with the test email exists
  - Firebase Auth: custom claims include `{ orgId: "chunk-3b-1-smoke-co", role: "owner", orgIds: ["chunk-3b-1-smoke-co"] }`

  - Each teammate gets a unique magic link printed

## E. Final Recommendation

**Would I personally allow a Customer Success person to provision Customer #2-#5 with the current 3B-1 surface, with no founder involvement on the org-provisioning step?**

**YES** — once the manual verification checklist passes on `peakops-internal-alpha`.

The CS person needs:
1. An ID token from a user with `peakopsInternalAdmin:true` (one-time setup, founder grants)
2. A copy of `scripts/activateCustomerOrg.cjs` (in the repo)
3. The customer's basics: name, industry, admin email, optional teammate list

They run the command, copy the magic link out of the output, paste into the welcome email template (also printed by the script), send to the customer. Done.
